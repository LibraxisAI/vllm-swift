## 2x2 analysis (answers to the framing questions)

### 1. Did vllm-metal numbers systematically jump vs README?

**No — they regressed**, and only on the small model. Pattern:

- **Qwen3-4B-4bit: vllm-metal essentially unchanged** (within ±3%). README's vllm-metal numbers for the 4B model are still accurate.
- **Qwen3-0.6B-4bit: vllm-metal regressed 13-44%**, worst at B=8 (-44%, 1170→652). Likely a fixed scheduling overhead that hurts tiny models proportionally more — at 4B the model compute dominates, at 0.6B the scheduler dominates.
- vllm-metal git log unchanged since 2026-04-23 (no recent commits). The regression may be a vLLM core version change (we're on 0.19.1 in the venv) rather than vllm-metal itself.

The "we matched/beat them by N×" claim in the README was correct **and got better** since publication.

### 2. Did vllm-swift numbers stay stable vs README?

**Yes.** Both models, all four concurrency levels, within ±3%. Only the bench script changed (subprocess-per-level for vllm-metal teardown), not the engine. vllm-swift numbers are reliable.

### 3. Where does paged actually need to win? B=64 specifically.

The headroom **at the paged-target workload (4B model, B=64)** is **143 tok/s, or 10.4%**. That's the absolute floor — paged needs to deliver more than just matching this gap.

The real wins paged unlocks aren't in tok/s on this 4B/B=64/short-context cell:

- **Long context**: at 4K+ context with B=32-64, contiguous BatchedKVCache hits memory-packing limits. Paged's block reuse and prefix sharing become decisive there. Need to add a 2K/4K/8K context column to the matrix to expose this gap.
- **Larger models at high concurrency**: 9B/27B at B=64 with mixed-length sequences. Contiguous cache wastes memory on padding to longest sequence. Paged packs at block granularity (16 tokens).
- **Prefix caching**: cross-request shared system prompts. Currently 0% reuse in vllm-swift. Paged with refcounted blocks unlocks this entirely.

**Paged target list:**
1. Match B=64 4B short-ctx within 5% (no regression)
2. Long-context B=32 (4K ctx): currently OOM-prone in contiguous, paged should run cleanly
3. Prefix caching shared-prompt workload: ≥2x speedup vs no caching

## Attention kernel microbench (vllm-metal, 2026-04-25)

`tools/benchmark/attention_benchmark.py --group decode`. Per-call ms, num_layers=1, block_size=16.

| Case | shape | v1 (paged) | v2 (paged) | sdpa (gather+SDPA) | gather overhead |
|------|-------|:---:|:---:|:---:|:---:|
| small | B=1, q=1, kv=128 | 0.41 | 0.34 | 0.39 | -5% |
| typical | B=8, q=1, kv=2048 | 0.37 | 0.39 | 0.82 | **2.2x slower** |
| big-head | B=8, q=1, kv=2048 | 0.64 | 0.70 | 1.04 | 1.6x slower |
| long | B=32, q=1, kv=8192 | 0.96 | 1.03 | 4.55 | **4.7x slower** |

**Implication:** naive "gather blocks → contiguous → SDPA" path (Phase 2 plan) is 4.7x slower than v1 at high concurrency / long context. Need real paged kernel for actual wins. Phase 2 ships as foundation only (not benchmarked).

**Targets for Phase 3 kernel:**
- Lower bound: match v1 within 30%
- Acceptable interim: match sdpa-gather within 10% (still 4.7x worse than v1, but ships paged scheduler)

### Implication

The story flips at 8K B=64. Below 8K, vllm-swift wins everywhere — even 4B/4K/B=64 by **+13%** vs vllm-metal. That's the contiguous-cache fast-path holding up well against paged at moderate context.

At 8K B=64, vllm-swift OOMs structurally:
- Per-layer BatchedKVCache `[64, 8, 8704, 128]` bf16 = ~1.06 GB
- × 36 layers (4B) × 2 (K and V) = ~76 GB just for the batched cache
- Plus per-request prefill caches (~57 GB before init_batched)
- Total exceeds 128 GB unified memory before decode starts

vllm-metal at 4B/8K/B=64 = **231 tok/s** by sharing block-granularity allocations across requests. That's the only paged-decisive cell in the matrix:

- **Phase 3 paged target: match vllm-metal 231 tok/s at 4B/8K/B=64, no OOM.** Anything ≥231 closes the gap; ≥260 (vllm-swift's 4B/4K/B=64 trajectory extrapolated) is a clean win.
- **Phase 3 secondary: hold the existing wins below 8K.** Don't regress 0.6B/2K/B=1 (309 → ?) or 4B/4K/B=64 (483 → ?). Paged kernel must beat or match short-context contiguous, not just long-context.
