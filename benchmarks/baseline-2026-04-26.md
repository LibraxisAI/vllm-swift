# Baseline benchmarks — 2026-04-26 (M5 Max 128GB)

Updates [baseline-2026-04-25.md](baseline-2026-04-25.md) with two methodology improvements and one capacity fix:

1. **Unique-prompts mode** (`--prompts unique`). The previous bench used `[prompt] * B` (B identical prompts), which let vllm-metal's prefix caching dedupe up to B-1 of B requests — measuring cache-hit speed for one engine and recompute speed for the other. This run uses B distinct prompts to compare raw compute fairly.
2. **Both engines report both metrics per cell.** `tps_decode = gen / decode_only_time` and `tps_e2e = gen / (prefill + decode_time)`. Comparing engines at different metrics (decode-only vs e2e) is meaningless at long context where prefill dominates.
3. **BatchedKVCache eager release fix** (`vllm-swift` commit `00e7538`, `mlx-swift-lm` commit `5d45da2`). Per-request `KVCacheSimple` instances are now released inside the `init_batched` copy loop. 4B/p8K/B=64 no longer OOMs.

The 2026-04-25 doc remains the date-locked snapshot of that day's measurements. This document supersedes its long-context cells.

## Methodology

- Script: `scripts/baseline_matrix.py` orchestrator runs `bench_throughput.py` (vllm-swift) and `bench_vllm_metal.py` (vllm-metal subprocess-per-level).
- 50 decode tokens, greedy (temp=0).
- 1 sample per cell. Median-of-3 still pending.
- Fresh subprocess per concurrency level (vllm-metal `EngineCore` does not reap on `del LLM` — see [#19849](https://github.com/vllm-project/vllm/issues/19849), [#1908](https://github.com/vllm-project/vllm/issues/1908), [#17273](https://github.com/vllm-project/vllm/issues/17273), [#24885](https://github.com/vllm-project/vllm/issues/24885)).
- Same model files for both engines. Same prompts (unique mode generates B distinct topics).
- vllm-metal venv at `~/.venv-vllm-metal/`.

At 18 tokens of actual GPU compute, the bookkeeping is the workload.

**Read:** decode advantage widens with both context length and concurrency. At 4B/4K/B=64 vllm-swift is **21× faster** at decode than vllm-metal under identical workloads with no prefix-cache help. Pattern matches the architectural diff: vllm-metal's paged decode pays per-step block-table indirection (128 blocks/request × 64 requests at 2K = ~8K block lookups per token); vllm-swift's contiguous BatchedKVCache reads a flat tensor.

E2E ratios at long context are narrower because vllm-swift's per-request sequential prefill is significantly slower than vllm-metal's batched prefill. At 4B/p4K/B=64: swift prefill 81s vs metal 135s — actually swift faster on raw prefill compute, but bridge bookkeeping eats most of the win.

## Capacity verification (post-fix)

Both 8K B=64 cells now run on vllm-swift after the BatchedKVCache eager-release fix. Captured separately from the long-ctx JSON.

| Cell | pre-fix swift | post-fix swift | metal |
|---|---|---|---|
| 0.6B / p=8K / B=64 | OOM | **runs (decode 432 tok/s)** | runs (decode 22 tok/s) |
| 4B / p=8K / B=64 | OOM | **runs (decode 259 tok/s)** | runs (decode 7 tok/s) |

Post-fix swift is **~20× faster** than vllm-metal at 0.6B/p=8K/B=64 and **~37× faster** at 4B/p=8K/B=64. The cell that previously OOMed for vllm-swift is now the cell where vllm-swift's decode lead is largest.

## Phase 3 paged target — reframed

The 2026-04-25 doc framed Phase 3 as "match vllm-metal 231 tok/s at 4B/p8K/B=64, no OOM." Both conditions are now met without paged work — eager release fix runs the cell at 259 tok/s.

Updated targets:

1. **Below 8K, no regression.** Paged kernel must hold or exceed contiguous BatchedKVCache decode at every measured cell. Current floor: 21× lead at 4B/p4K/B=64 must not regress.
2. **Variable-length packing.** The remaining capacity advantage of paged is fitting mixed-length sequences efficiently. Contiguous cache pads to longest sequence per slot; paged shares blocks at 16-token granularity. Test cell: B=64 with prompts ranging 256-8192 tokens, measure peak memory and effective concurrency.
3. **Prefix caching infrastructure.** Block-hash → KV slice lookup is natural over paged storage. The user-visible feature is what matters; paged is the substrate. Target: 2× speedup on shared system-prompt workloads.

The "compute parity" framing is closed. Future paged work is feature-driven (variable-length packing, prefix caching), not gap-closing.
